# linker-lang v0.2.0 — Core

**The first release with domain logic.** v0.2.0 turns the v0.1.0 scaffold into a working linker: it combines independently compiled objects into a single laid-out image — merging sections, resolving symbols, and patching relocations. The release introduces the whole public surface — the `Object` input model, the `Linker` and the `link` shortcut, and the `Image` they produce — together with full rustdoc, an `API.md` reference, property tests, and Criterion benchmarks. The surface remains pre-1.0 and will be frozen at `1.0.0`.

## What is linker-lang?

A small static linker for the back end of a language toolchain. A compiler emits each translation unit on its own — a few named byte sections, some symbols marking addresses inside them, and relocations that still need an address filled in. linker-lang joins those islands: it lays the sections of every object out end to end, resolves each symbol to its final address, and patches the relocations with the addresses they were waiting for. The model is format-agnostic — an ELF or PE reader feeds the same `Object`, and a writer turns an `Image` back into a file.

## What's new in 0.2.0

### The `Object` input model

An `Object` is one compilation unit: named byte sections, the symbols defined in them, and the relocations to patch. It is built with a handful of methods and carries no addresses of its own — the linker assigns them. Sections are addressed by name, and appending to a name that exists extends that section:

```rust
use linker_lang::{Object, Width};

let mut obj = Object::new("module.o");
obj.section(".text", vec![0u8; 16]);
obj.define("main", ".text", 0);
obj.define("helper", ".text", 8);

// A `.data` slot that should hold the address of `helper`.
obj.section(".data", vec![0u8; 8]);
obj.relocate(".data", 0, "helper", Width::U64, 0);
```

`Width` (`U32` / `U64`) is the size of the address a relocation patches in, written little-endian.

### The `Linker` and the `link` shortcut

`link` is the one-call entry point; `Linker` is the same thing with a base address and an entry point to configure. A link is a few linear passes — merge sections, resolve symbols, patch relocations — and the result is an `Image`:

```rust
use linker_lang::{Linker, Object, Width};

let mut code = Object::new("code");
code.section(".text", [0u8; 8]);
code.define("on_start", ".text", 0);

let mut table = Object::new("table");
table.section(".data", [0u8; 8]);
table.relocate(".data", 0, "on_start", Width::U64, 0);

let image = Linker::new()
    .base_address(0x40_0000)
    .entry("on_start")
    .link(&[code, table])
    .unwrap();

assert_eq!(image.entry(), image.symbol("on_start"));
let slot = image.section(".data").unwrap().data();
assert_eq!(u64::from_le_bytes(slot.try_into().unwrap()), image.symbol("on_start").unwrap());
```

### The `Image` and its link map

An `Image` holds the laid-out sections, the resolved symbol table, and the entry-point address. Sections are addressed by `symbol(name)`, bytes are read with `section(name)`, and the `Display` impl renders a readable link map:

```text
entry = 0x0000000000400000
.text @ 0x0000000000400000 (8 bytes)
.data @ 0x0000000000400008 (8 bytes)
symbols:
    0x0000000000400000 on_start
```

Each output section's bytes are final — relocations into it have been patched — so the image is ready to be written to a file or loaded at its address.

### Errors at the boundary

Every way a link can fail is a distinct `LinkError` variant that names the object, symbol, or section involved: a symbol defined by two objects, a relocation or entry point with no definition, a symbol or relocation in a section the object never created, a slot that runs past its section, an address too large for its width, or a layout that exceeds the address space. The linker reports the reason rather than producing a partial or wrong image:

```rust
use linker_lang::{link, LinkError, Object, Width};

let mut obj = Object::new("uses.o");
obj.section(".data", [0u8; 8]);
obj.relocate(".data", 0, "external", Width::U64, 0); // never defined

assert!(matches!(link(&[obj]), Err(LinkError::UndefinedSymbol { .. })));
```

`LinkError` is `#[non_exhaustive]` (a future relocation kind or layout constraint may add a variant) and implements `Display` and `std::error::Error`.

### `no_std` and `serde`

The crate is `#![no_std]`-capable: with `default-features = false` it needs only `alloc`. The optional `serde` feature derives `Serialize` / `Deserialize` for `Image`, `OutputSection`, `Width`, and `LinkError`, so a linked image or a failure can be cached, logged, or moved between tools.

### Tests, examples, and benchmarks

The link workflow is exercised end to end. `tests/workflow.rs` compiles functions with `codegen-lang`, wraps each as a linkable object, and links them — symbol resolution, section merging, and relocation patching on real backend output. `tests/invalid.rs` confirms every failure is reported as the precise error, and `tests/properties.rs` generates random links and checks symbol addresses and patched slots against an independent computation. Two runnable examples — `link_map` (hand-built objects) and `from_codegen` (the compiler pipeline) — and Criterion benchmarks for the symbol-resolution and section-merge cost drivers round out the release.

## Breaking changes

**None.**

v0.1.0 had no domain logic, so the entire public surface is new and additive.
