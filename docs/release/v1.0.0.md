# linker-lang v1.0.0 — stable

**1.0.** The public API is frozen and stable. v0.1.0 stood up the scaffold and the quality gates; v0.2.0 built the core — merging sections, resolving symbols, and patching relocations into a single laid-out image. This release ratifies that surface as the `1.0` contract: it follows Semantic Versioning, carries no breaking changes before `2.0`, and ships with the full test and benchmark suite verified on Linux, macOS, and Windows.

There is no behavioural change from `0.2.0` — a set of objects that linked then links identically now. The release adds the stability documentation and one MSRV-only lint fix.

## What is linker-lang?

A small static linker for the back end of a language toolchain. A compiler emits each translation unit on its own — a few named byte sections, some symbols marking addresses inside them, and relocations that still need an address filled in. linker-lang joins those islands: it lays the sections of every object out end to end, resolves each symbol to its final address, and patches the relocations with the addresses they were waiting for. The model is format-agnostic — an ELF or PE reader feeds the same `Object`, and a writer turns an `Image` back into a file.

```rust
use linker_lang::{link, Object, Width};

// The code: a symbol `start` at the top of `.text`.
let mut code = Object::new("code");
code.section(".text", [0x90, 0x90, 0x90, 0x90]);
code.define("start", ".text", 0);

// The data: an 8-byte slot that should hold the address of `start`.
let mut data = Object::new("data");
data.section(".data", [0u8; 8]);
data.relocate(".data", 0, "start", Width::U64, 0);

let image = link(&[code, data]).expect("symbols resolve");
assert_eq!(image.symbol("start"), Some(0));
let slot = image.section(".data").unwrap().data();
assert_eq!(u64::from_le_bytes(slot.try_into().unwrap()), 0); // patched with start's address
```

## The 1.0 surface

The frozen contract:

- **Input** — the `Object` model: named byte `section`s, the `define`d symbols inside them, and the `relocate`d holes to fill, plus `Width` (`U32` / `U64`) for the address a relocation patches.
- **Linking** — the `Linker` (a base address and an optional entry point) and the `link` shortcut for the default configuration.
- **Output** — the `Image` (laid-out sections, the resolved symbol table, the entry point, and a link-map `Display`) and its `OutputSection`s (name, address, final relocated bytes).
- **Errors** — `LinkError`, naming the object, symbol, or section involved in each failure.
- **Serialization** — `serde` for `Image`, `OutputSection`, `Width`, and `LinkError` under the `serde` feature.

`LinkError` is `#[non_exhaustive]`, so a future relocation kind or layout constraint that adds a failure reason is a non-breaking addition.

## SemVer promise

The crate follows [Semantic Versioning](https://semver.org):

- No documented item is removed or changed in a breaking way within `1.x`; breaking changes wait for `2.0`.
- New functionality is additive and arrives in minor releases. A new `LinkError` variant is a minor change.
- The MSRV is Rust `1.85`; raising it is a minor change, never a patch.
- Behaviour is part of the contract: a set of objects that links today keeps linking, symbol addresses are laid out the same way, and the link-map `Display` form is stable for a given image.

The full statement lives in [`docs/API.md`](https://github.com/jamesgober/linker-lang/blob/main/docs/API.md#semver-promise).

## Breaking changes

**None.** `1.0.0` makes no breaking change to the `0.2.0` surface.

## Hardening since 0.2.0

- **Stability documentation.** The crate root, `README`, and `docs/API.md` now state the frozen `1.0` surface and the SemVer promise.
- **MSRV lint fix.** A redundant lifetime in `link.rs` that older (`1.85`) `clippy` flagged as `needless_lifetimes` — newer `clippy` does not — was elided, so the 1.85 CI jobs lint clean. No API or behavioural change.

The runnable examples (`link_map`, `from_codegen`), the `serde` round-trip tests, and the property tests shipped with `0.2.0` and carry forward unchanged.

## Verification

Run on Windows x86_64 with WSL2 (Ubuntu), Rust stable 1.95.x and the 1.85 MSRV toolchain; the same commands run in the CI matrix across Linux, macOS, and Windows:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo +1.85 clippy --all-targets --all-features -- -D warnings   # MSRV lint
cargo test
cargo test --all-features
cargo test --no-default-features          # no_std build
cargo run --example link_map
cargo run --example from_codegen
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo bench --bench bench --no-run
cargo +1.85 build --all-features          # MSRV
cargo deny check
cargo audit
```

All green. Test counts at this tag:

- Default features: 27 unit + 14 integration (4 workflow + 7 invalid + 3 property) + 19 doctests.
- `--all-features`: the above + 2 `serde` round-trip tests.
- Two runnable examples (`cargo run --example link_map`, `cargo run --example from_codegen`).

## What's next

The `1.0` surface is stable until `2.0`. Future work is documentation, tests, and internal optimization that does not change the public contract — plus the additive room the design leaves: section alignment and a base-per-section layout, PC-relative relocation kinds, and ELF or PE readers and writers on either side of the same `Object` and `Image`, each a minor release when it lands.

## Installation

```toml
[dependencies]
linker-lang = "1"

# no_std:
linker-lang = { version = "1", default-features = false }

# with serialization:
linker-lang = { version = "1", features = ["serde"] }
```
